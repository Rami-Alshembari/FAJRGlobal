Implementation Notes

Data sources, unification approach, and extension path.

The master dashboard uses a shared design system and centralized configuration to present three existing dashboards as one reporting platform while preserving the original detailed analytics in embedded module views.

What Changed

System-wide unification

New master entry point: `index.html` now acts as the executive homepage for all Gaza reporting streams.
Shared design language: typography, spacing, cards, buttons, panels, and page shells are standardized across the new portal.
Program modules: each source dashboard now has a dedicated page inside the portal with unified context, KPIs, narrative, and embedded detail view.
Centralized data layer: `assets/js/config.js` stores program metadata, headline KPIs, notes, and chart definitions in one place.

Navigation Logic

How movement works

Top navigation: persistent access to overview, each program module, and methodology.
Overview cards: each program card links directly into its module page.
Program pages: each module includes a contextual summary above the preserved legacy dashboard for deeper analysis.
Back to overview: every module page keeps a return action in the header.

Style Unification

How consistency is applied

Color system: white and stone surfaces, navy text, and restrained teal and gold accents.
Type hierarchy: Fraunces is used for strategic headings and Manrope for interface and analytical text.
Interaction layer: shared hover states, subtle elevation, responsive grids, and KPI count-up animations are handled by shared assets.
Legacy preservation: the source dashboards remain intact and are surfaced inside a controlled frame shell rather than rewritten destructively.

How To Extend

Adding future dashboards

Step 1: add a new program object to `assets/js/config.js` with title, metrics, narrative, and legacy source path.
Step 2: create a new page in `pages/` using the same shell pattern and set `data-program` to the new key.
Step 3: add the page to the site navigation and overview module cards.
Step 4: if deeper integration is needed later, replace the legacy iframe with a fully refactored module while keeping the shared shell.
